# ytm-sync

🇷🇺 Русский (основной) · [🇬🇧 English](README.en.md)

Автоматическая синхронизация **публичного плейлиста YouTube Music** в медиатеку
**Navidrome**. Новые треки, которые ты добавляешь в плейлист, по расписанию
скачиваются (оригинальный `opus`/`m4a`, с обложкой и тегами), сводятся в один
альбом, чтобы не засорять библиотеку, и отдаются в Navidrome и как файлы, и как
упорядоченный `.m3u`-плейлист.

Работает как один сервис Docker Compose рядом с контейнером Navidrome.

> 📖 **Подробный пошаговый гайд** (установка, как всё устроено, обслуживание,
> решение проблем): [`docs/SETUP.md`](docs/SETUP.md).

---

## Почему именно так

Стабильно вытащить большой плейлист YouTube Music сложнее, чем кажется. Здесь
сознательный «раздельный» подход, выстраданный методом проб:

| Задача | Решение | Причина |
|---|---|---|
| **Получить полный список треков** | **YouTube Data API v3** (ключ) | Анонимно большой плейлист обрезается до ~100. Cookies работают, но **протухают за часы**, если аккаунт используется ещё где-то. Ключ не ротируется. |
| **Скачать аудио** | `yt-dlp`, клиент `android_vr`, **без cookies** | Этот клиент отдаёт прямые аудио-URL без JS-челленджа «n-signature». С cookies web/TV-клиенты падают с *«Only images available»*. |
| **Возрастные треки** | `yt-dlp[default]` (тянет `yt-dlp-ejs`) + `deno` + cookies | Решает JS-челлендж, и возрастные видео отдают форматы. |
| **Структура библиотеки** | Все треки тегируются `ALBUM=YTM`, `ALBUMARTIST=…` | Иначе каждый сингл становится отдельным альбомом и забивает экран *Albums*. |
| **Порядок плейлиста** | Генерируемый `.m3u`, перестраивается каждый прогон | Navidrome импортирует его как плейлист в точном порядке. |
| **Не гадить на локальный диск** | Опциональный guard сетевой шары | Если шара-назначение отвалилась — прогон прерывается, а не пишет молча на локальный диск. |

## Возможности

- ⏱️ Периодическая синхронизация (по умолчанию раз в 4ч) + прогон сразу при старте.
- 🎵 Оригинальное аудио (без перекодирования), встроенные теги и обложка.
- 📃 Упорядоченный `.m3u`, который Navidrome импортирует как плейлист.
- 🗂️ Сведение в один альбом + номера треков по порядку плейлиста — библиотека
  чистая, а альбом открывается в порядке плейлиста (свежее сверху).
- ➕ Режим только добавления: удаление трека из плейлиста не удаляет локальный файл.
- ♻️ Самовосстановление: плейлист строится из реально лежащих на диске файлов
  (videoId читается из вшитого тега `purl`).
- 🛟 Инструмент добора: вытягивает возрастные и ищет замены для удалённых/заблокированных.
- 💾 Переключаемое хранилище: сетевая шара / локальный диск, опц. зеркало в S3 через `rclone`.
- 🔁 Авто-восстановление: если шара ещё не примонтирована (например, после ребута) —
  контейнер ждёт и подхватывает её без перезапуска.

## Требования

- Docker + Docker Compose.
- Navidrome, смотрящий в папку с музыкой, в которую ты можешь и писать.
- Ключ **YouTube Data API v3** (бесплатный) — см. ниже.

## Быстрый старт

```bash
git clone <этот-репозиторий> ytm-sync
cd ytm-sync
cp .env.example .env
# отредактируй .env — впиши PLAYLIST_URL и YT_API_KEY
docker compose up -d --build
docker compose logs -f
```

### Получить ключ YouTube Data API (≈3 минуты, бесплатно)

1. Открой <https://console.cloud.google.com/> и создай (или выбери) проект.
2. **APIs & Services → Library** → включи **«YouTube Data API v3»**
   (прямая ссылка: <https://console.cloud.google.com/apis/library/youtube.googleapis.com>).
3. **APIs & Services → Credentials → Create credentials → API key**.
4. Скопируй ключ (`AIza…`) в `YT_API_KEY` в `.env`.

Квота копеечная: листинг плейлиста на 650 треков стоит ~13 единиц при лимите 10 000/день.

> Нет ключа? Используется резерв через `yt-dlp` + опциональный `data/cookies.txt`.
> Без cookies большой плейлист режется до ~100 треков — поэтому ключ рекомендуется.

## Конфигурация

Всё настраивается через `.env` (см. [`.env.example`](.env.example)):

| Переменная | По умолчанию | Описание |
|---|---|---|
| `PLAYLIST_URL` | — | URL публичного плейлиста YouTube Music (обязательно). |
| `YT_API_KEY` | — | Ключ YouTube Data API v3 (рекомендуется). |
| `INTERVAL_SECONDS` | `14400` | Как часто проверять плейлист (4ч). |
| `MUSIC_SUBDIR` | `YTM` | Подпапка внутри корня музыки для скачанного. |
| `ALBUM_NAME` | `YTM` | Тег альбома для всех треков (группировка). |
| `ALBUM_ARTIST` | `YouTube Music` | Тег album-artist для всех треков. |
| `MUSIC_HOST_PATH` | `/mnt/Music` | Путь на хосте, монтируемый в контейнер как `/music`. |
| `REQUIRE_SHARE` | `true` | Guard: писать только если `/music` — живая сетевая шара. Для локального диска `false`. |
| `RCLONE_REMOTE` | — | Опц. rclone-remote для зеркала, напр. `s3:bucket/music`. |
| `TZ` | `Europe/Moscow` | Таймзона. |

### Режимы хранилища

- **Сетевая шара (по умолчанию):** монтируешь CIFS/NFS-шару в `MUSIC_HOST_PATH`,
  оставляешь `REQUIRE_SHARE=true`. Если шара пропала — прогон прерывается (локально
  не пишет), а контейнер сам ждёт её возвращения.
- **Локальный диск:** ставишь `MUSIC_HOST_PATH` на локальную папку и `REQUIRE_SHARE=false`.
- **S3 / удалённое (опц. зеркало):** кладёшь `rclone.conf` в `./data/` и ставишь
  `RCLONE_REMOTE=s3:твой-бакет/music`. После каждой синхронизации новые файлы
  копируются туда через `rclone copy`. (Отдача бакета в Navidrome — отдельная задача,
  например `rclone mount`.)

## Как идёт прогон

1. Получить упорядоченный список videoId (YouTube Data API или резерв yt-dlp).
2. Скачать новое через `yt-dlp` (`android_vr`, archive исключает повторы), пропуская
   заведомо недоступные id.
3. Проставить теги одного альбома на ещё неразмеченные файлы.
4. Пересобрать карту `videoId → файл` с диска (читая вшитые теги `purl`).
5. Опционально — зеркалировать на remote через `rclone`.
6. Перезаписать `.m3u` в текущем порядке плейлиста (с учётом замен).

## Добор недостающих — `scripts/backfill.py`

Часть позиций падает при обычном прогоне: возрастные, удалённые, региональные,
только-для-Premium. Скрипт добора для каждой недостающей пробует:

1. скачать **оригинал** (cookies + EJS-решатель — лечит возрастные);
2. иначе **найти на YouTube** ту же песню по названию и скачать замену, записав её
   в `data/replacements.tsv`, чтобы `.m3u` всё равно ссылался на неё;
3. иначе пометить в `data/unavailable.txt`, чтобы будущие прогоны её пропускали.

```bash
# все недостающие
docker run --rm \
  -v "$PWD/data:/config" -v /mnt/Music:/music \
  -v "$PWD/.env:/env:ro" -v "$PWD/scripts/backfill.py:/backfill.py:ro" \
  --entrypoint python3 ytm-sync:latest /backfill.py

# только конкретные videoId (форс-ретрай)
... /backfill.py VIDEOID1 VIDEOID2
```

Для возрастных нужен валидный `data/cookies.txt` (формат Netscape). Экспортируй его из
браузера, где ты залогинен — надёжнее всего Firefox
(`yt-dlp --cookies-from-browser firefox --cookies cookies.txt <любой-url>`).

## Файлы состояния (`./data`, монтируется в `/config`)

| Файл | Назначение |
|---|---|
| `archive.txt` | download-archive yt-dlp (уже скачанные videoId). |
| `idmap.tsv` | `videoId → относительный путь`, перестраивается с диска. |
| `replacements.tsv` | `origId → путь файла-замены` (для `.m3u`). |
| `unavailable.txt` | id, которые больше не пробуем (удалён/заблокирован/private). |
| `cookies.txt` | Опц. Резерв для листинга и возрастных. |
| `rclone.conf` | Опц. Используется, если задан `RCLONE_REMOTE`. |

## Решение проблем

- **Плейлист обрезается до ~100** → задай `YT_API_KEY` или обнови `cookies.txt`.
- **`Only images available` / `Requested format is not available`** → это nsig-челлендж;
  собери образ с `yt-dlp[default]` и `deno`.
- **Ничего не качается, прогон прерывается** → при `REQUIRE_SHARE=true` шара не
  примонтирована; проверь монтирование на хосте (и маркер `data/.ytm_share_ok` на шаре).
- **yt-dlp внезапно сломался** → он сам обновляется при старте; `docker compose restart`.
