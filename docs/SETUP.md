# Полный гайд по установке и работе ytm-sync

🇷🇺 Русский · [🇬🇧 English](SETUP.en.md)

Этот документ — подробная пошаговая инструкция: от нуля до работающей
синхронизации плейлиста YouTube Music в Navidrome, плюс разбор как всё устроено
внутри и как это обслуживать. Если нужен короткий обзор — смотри
[README](../README.md).

---

## 0. Что это и зачем

`ytm-sync` берёт **публичный плейлист YouTube Music**, скачивает из него треки в
папку с музыкой Navidrome и держит её в актуальном состоянии: добавил трек в
плейлист — через несколько часов он сам появился в Navidrome (и в любом клиенте
вроде Amperfy/DSub).

Что получаешь:
- оригинальное аудио (`opus`/`m4a`, без перекодирования) с обложкой и тегами;
- один аккуратный альбом в библиотеке вместо сотен «одиночек»;
- плейлист в Navidrome в том же порядке, что и в YouTube Music;
- автоматику: проверка по расписанию, запуск при старте, само-обновление yt-dlp.

---

## 1. Что нужно заранее

| Компонент | Зачем | Проверка |
|---|---|---|
| Docker + Docker Compose | в нём крутится сервис | `docker --version && docker compose version` |
| Navidrome | музыкальный сервер | уже развёрнут и смотрит в папку с музыкой |
| Папка с музыкой **на запись** | туда складываются треки | та же, что отдаёт Navidrome, но доступная на запись |
| Ключ YouTube Data API v3 | получать полный список плейлиста | бесплатно, см. шаг 3 |

> Navidrome у тебя смотрит в папку (часто `/mnt/Music`) только на чтение
> (`:ro`) — это нормально. `ytm-sync` пишет в ту же папку, но смонтированную на
> запись отдельно. Navidrome потом сам её сканирует.

## 2. Получить файлы проекта

```bash
git clone https://github.com/Laur1dor/ytm-nd.git ytm-sync
cd ytm-sync
```

Структура:

```
ytm-sync/
├─ compose.yaml        # описание docker-сервиса
├─ Dockerfile          # образ: python + ffmpeg + yt-dlp[default] + deno + rclone
├─ entrypoint.sh       # цикл: обновить yt-dlp, дождаться шары, синк, спать
├─ sync.py             # вся логика синхронизации
├─ scripts/backfill.py # добор недостающих треков
├─ .env.example        # шаблon конфигурации
└─ docs/               # эта документация
```

## 3. Ключ YouTube Data API v3 (≈3 минуты, бесплатно)

Зачем именно ключ, а не cookies: анонимно YouTube отдаёт у большого плейлиста
только ~100 треков, а cookies аккаунта **протухают за часы**. Ключ API не
протухает и отдаёт весь список.

1. Открой <https://console.cloud.google.com/> и создай проект (кнопка вверху →
   *New Project*). Можно использовать любой существующий.
2. Включи API: **APIs & Services → Library**, найди **«YouTube Data API v3»**,
   нажми **Enable**. Прямая ссылка:
   <https://console.cloud.google.com/apis/library/youtube.googleapis.com>.
   > Если на шаге создания ключа список «API restrictions» пустой — значит этот
   > пункт пропущен: сначала **Enable**, потом ключ.
3. Создай ключ: **APIs & Services → Credentials → Create credentials → API key**.
4. Скопируй ключ вида `AIza...`. По желанию в настройках ключа ограничь его
   только этим API (Application restrictions: None, API restrictions: YouTube
   Data API v3).

Квота: листинг плейлиста на 650 треков ≈ 13 единиц при лимите 10 000/день —
то есть фактически бесконечно.

## 4. Настроить `.env`

```bash
cp .env.example .env
nano .env
```

Минимально нужно заполнить `PLAYLIST_URL` и `YT_API_KEY`. Все параметры:

| Переменная | По умолчанию | Что делает |
|---|---|---|
| `PLAYLIST_URL` | — | URL публичного плейлиста YTM (`https://music.youtube.com/playlist?list=...`). |
| `YT_API_KEY` | — | Ключ из шага 3. |
| `INTERVAL_SECONDS` | `14400` | Период проверки плейлиста (сек). 14400 = 4 часа. |
| `MUSIC_SUBDIR` | `YTM` | Подпапка внутри музыкального корня для треков. |
| `ALBUM_NAME` | `YTM` | Имя альбома-обёртки для всех треков. |
| `ALBUM_ARTIST` | `YouTube Music` | Album-artist для всех треков. |
| `MUSIC_HOST_PATH` | `/mnt/Music` | Путь к музыке на **хосте**, монтируется в `/music`. |
| `REQUIRE_SHARE` | `true` | Гард: писать только на живую сетевую шару (см. шаг 5). |
| `RCLONE_REMOTE` | — | Опц. зеркало в S3/удалённое хранилище (см. шаг 5). |
| `TZ` | `Europe/Moscow` | Таймзона для логов/расписания. |

Как получить `PLAYLIST_URL`: в YouTube Music открой плейлист → «Поделиться» →
«Копировать ссылку». Плейлист должен быть **публичным** (или «по ссылке»).

## 5. Выбрать режим хранилища

**Вариант А — сетевая шара (по умолчанию, рекомендуется для NAS).**
Музыка лежит на CIFS/NFS-шаре, смонтированной на хосте (напр. `/mnt/Music`).
Оставь `REQUIRE_SHARE=true`. Перед каждой записью сервис проверяет, что `/music`
— это живая сетевая ФС и на ней есть файл-маркер `.ytm_share_ok`. Если шара
отвалилась — прогон **прерывается, и на локальный диск ничего не пишется**.
Создай маркер один раз:

```bash
touch /mnt/Music/.ytm_share_ok
```

**Вариант Б — локальный диск.**
Поставь `MUSIC_HOST_PATH=/путь/к/локальной/папке` и `REQUIRE_SHARE=false`.

**Вариант В — зеркало в S3/удалённое (поверх А или Б).**
Положи конфиг rclone в `./data/rclone.conf` и задай `RCLONE_REMOTE=s3:bucket/music`.
После каждого синка новые файлы копируются туда (`rclone copy`). Отдачу этого
бакета в Navidrome настраивай отдельно (например `rclone mount`).

---

## 6. Собрать и запустить

```bash
docker compose up -d --build
docker compose logs -f
```

Первый прогон скачает весь плейлист (для 600+ треков это ~1.5–2 часа, идёт в
фоне). Дальше — каждые `INTERVAL_SECONDS` подтягиваются только новые.

Полезные команды:

```bash
docker compose logs -f ytm-sync     # смотреть логи
docker compose restart              # форс-прогон сейчас (синк идёт сразу при старте)
docker compose up -d                # применить изменения .env
docker compose down                 # остановить
```

## 7. Проверить в Navidrome

Navidrome сканирует библиотеку по своему расписанию (`ND_SCANSCHEDULE`, по
умолчанию у многих 1ч). Чтобы увидеть сразу — можно перезапустить контейнер
Navidrome (`docker restart navidrome`), он сканирует при старте.

Появится:
- **Альбом** с именем из `ALBUM_NAME` (по умолчанию «YTM») — все треки в одном
  альбоме, отсортированы в порядке плейлиста (свежие сверху).
- **Плейлист** с именем подпапки (по умолчанию «YTM») — Navidrome авто-импортирует
  файл `<MUSIC_SUBDIR>.m3u` в порядке плейлиста.

## 8. Как это работает внутри

Каждый прогон (`sync.py`):

1. **Список треков.** Берёт упорядоченный список videoId через YouTube Data API
   (резерв — `yt-dlp --flat-playlist` + cookies). Порядок = порядок плейлиста.
2. **Скачивание.** `yt-dlp` клиентом `android_vr` **без cookies** качает новые
   треки. `--download-archive` исключает повторные скачивания. Заведомо
   недоступные id (`data/unavailable.txt`) пропускаются.
3. **Сверка карты.** Для каждого файла на диске читается вшитый тег `purl`
   (ссылка на видео) → строится карта `videoId → файл`. Поэтому плейлист
   собирается из того, что **реально лежит на диске** (само-восстановление).
4. **Теги.** На все файлы проставляется единый альбом и **номер трека = позиция
   в плейлисте** (чтобы и альбом открывался в порядке плейлиста). Пишутся только
   изменившиеся файлы.
5. **Зеркало.** Если задан `RCLONE_REMOTE` — `rclone copy` в удалённое хранилище.
6. **Плейлист.** Перезаписывается `<MUSIC_SUBDIR>.m3u` в порядке плейлиста; файлы
   в папке, которых нет в плейлисте, дописываются в конец.

### Почему такие решения

| Решение | Причина |
|---|---|
| Список — через API, не cookies | анонимно плейлист режется до ~100; cookies протухают за часы |
| Скачивание — `android_vr` без cookies | отдаёт прямые аудио-URL без JS-челленджа «n-signature»; с cookies web/tv-клиенты падают «Only images available» |
| Возрастные — `yt-dlp[default]` + `deno` + cookies | пакет `yt-dlp-ejs` решает JS-челлендж |
| Один альбom + номера треков | иначе сотни одиночных альбомов; номера дают порядок плейлиста в альбоме |
| Гард сетевой шары | не писать на локальный диск, если шара отвалилась |

## 9. Добор недостающих треков — `scripts/backfill.py`

Часть позиций падает на обычном прогоне: **возрастные**, **удалённые**,
**региональные**, **только-для-Premium**. Скрипт добора для каждой недостающей:

1. пробует скачать **оригинал** (cookies + EJS-решатель — лечит возрастные);
2. иначе **ищет на YouTube** ту же песню по названию (`ytsearch`) и качает
   замену, записывая её в `data/replacements.tsv` (плейлист всё равно
   указывает на неё на правильной позиции);
3. иначе помечает в `data/unavailable.txt`, чтобы будущие прогоны не долбились.

```bash
# все недостающие
docker run --rm \
  -v "$PWD/data:/config" -v /mnt/Music:/music \
  -v "$PWD/.env:/env:ro" -v "$PWD/scripts/backfill.py:/backfill.py:ro" \
  --entrypoint python3 ytm-sync:latest /backfill.py

# только конкретные videoId (форс-ретрай)
... /backfill.py VIDEOID1 VIDEOID2
```

Для возрастных нужен валидный `data/cookies.txt`.

## 10. Cookies (когда и зачем)

Cookies нужны **только** в двух случаях: (а) нет API-ключа (резервный листинг),
(б) добор возрастных треков. Для обычной работы они не нужны.

Экспорт (надёжнее всего из Firefox, где ты залогинен в YouTube):

```bash
yt-dlp --cookies-from-browser firefox --cookies cookies.txt "https://www.youtube.com/watch?v=dQw4w9WgXcQ"
cp cookies.txt ./data/cookies.txt
```

> Cookies YouTube ротируются и быстро устаревают, особенно если этим аккаунтом
> активно пользуешься в браузере. Поэтому для листинга используется API-ключ, а
> cookies — только разово под добор.

## 11. Обслуживание

- **yt-dlp** сам обновляется при старте контейнера и раз в сутки. Если YouTube
  что-то сломал — `docker compose restart`.
- **Поменять плейлист/интервал/папку** — правишь `.env`, затем `docker compose up -d`.
- **Скрипты** `sync.py`/`entrypoint.sh` смонтированы как volume — правки
  применяются перезапуском без пересборки. Изменения `Dockerfile` требуют
  `docker compose build`.

## 12. Решение проблем (FAQ)

| Симптом | Причина / решение |
|---|---|
| В плейлисте только ~100 треков | не задан `YT_API_KEY` (или невалиден); проверь ключ |
| `Only images available` / `Requested format is not available` | nsig-челлендж; пересобери образ — нужны `yt-dlp[default]` и `deno` |
| Прогон сразу прерывается, ничего не качает | `REQUIRE_SHARE=true`, а шара не примонтирована; проверь хост-маунт и маркер `.ytm_share_ok` |
| Возрастной трек не качается | нужен свежий `data/cookies.txt`; затем `backfill.py <id>` |
| Альбом сортируется по алфавиту, а не по плейлисту | старые файлы без номеров; дождись прогона (он проставит) или `docker compose restart` |
| В Navidrome два одинаковых плейлиста | дубликат от частых пересканов; удали лишний в БД/через UI |
| Новые треки не появляются | проверь логи (`docker compose logs`), что список приходит полный и идёт скачивание |

## 13. Файлы состояния (`./data` → `/config`)

| Файл | Назначение |
|---|---|
| `archive.txt` | download-archive yt-dlp (что уже скачано) |
| `idmap.tsv` | `videoId → относительный путь файла` (строится с диска) |
| `replacements.tsv` | `origId → путь файла-замены` (для `.m3u`) |
| `unavailable.txt` | id, которые больше не пробуем |
| `cookies.txt` | опц., резерв листинга и возрастные |
| `rclone.conf` | опц., если задан `RCLONE_REMOTE` |

## 14. Бонус-инструменты (`scripts/`)

- **`import_radio.py`** — заливает в Navidrome большой набор интернет-радиостанций
  из radio-browser.info, сгруппированных по категориям (категория зашивается в имя
  «🎧 Электроника · …», чтобы группировались при сортировке — своих папок для радио
  в Navidrome нет). Берёт только https-потоки (http блокируется mixed-content в
  HTTPS-вебе и на iOS; http→https проверяется реально). Запуск — при остановленном
  Navidrome, с бэкапом БД (см. шапку файла). Лимиты/жанры правятся в начале файла.

- **`nas-share-watch.sh`** — для сетапа, где музыка на сетевой шаре, которая
  поднимается после ребута с задержкой. systemd-сервис в системе с Docker: при
  старте ждёт появления шары (по маркеру) и перезапускает контейнеры, которым она
  нужна (например `navidrome`), — чтобы не делать это вручную.
